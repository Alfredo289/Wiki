Summary

Claude-Mem is a persistent memory compression system for Claude Code that captures what Claude learns during tool interactions, compresses observations into semantic summaries, and makes them available to future sessions. It addresses context-compression through progressive disclosure rather than file compression.

Architecture

  • Lifecycle hooks tie into Claude Code events: SessionStart, UserPromptSubmit, PostToolUse, Stop, SessionEnd
  • Worker service runs on port 37777 with HTTP API and web viewer UI (Bun runtime)
  • Storage uses SQLite with FTS5 full-text search plus Chroma vector database for hybrid semantic/keyword search

3-Layer Progressive Disclosure

The key design: instead of dumping all context at once, retrieval happens in three layers with ~10x token savings:

  1. search - compact index with IDs (~50-100 tokens per result)
  2. timeline - chronological context around observations
  3. get_observations - full details for filtered IDs (~500-1,000 tokens per result)

This lets Claude selectively load only the context it needs rather than consuming tokens on irrelevant history.

Relationship to Other Tools

Solves the same problem as Caveman Compress (reducing token costs for persistent context) but from the opposite direction: claude-mem adds a retrieval layer so you don’t load everything, while caveman compresses the files themselves. Both are context-compression strategies.

Also related to compound-engineering: claude-mem is infrastructure for accumulating project knowledge over time, which is the core promise of compound engineering.

Advanced Features

  • Endless Mode - biomimetic memory for extended sessions
  • Privacy controls - <private> tags exclude sensitive content
  • Real-time feeds - optional streaming to Telegram, Discord, Slack
  • Cross-platform: supports Claude Code, Gemini CLI, OpenClaw

Counter-arguments

  • Running a local HTTP service + SQLite + Chroma is heavy infrastructure for memory management
  • Lifecycle hooks add latency to every tool interaction
  • 10x token savings claim depends on how selectively the agent actually retrieves - a lazy agent might still pull everything
  • AGPL-3.0 license limits commercial use

Data gaps

  • No benchmarks showing actual token savings in real workflows
  • No comparison with simpler approaches (e.g., just a well-maintained CLAUDE.md)
  • Unclear how well the vector search handles cross-session concept drift
  • No data on storage growth rate over long-running projects
