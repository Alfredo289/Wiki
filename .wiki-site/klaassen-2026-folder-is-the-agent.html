Source: Article by Kieran Klaassen, Every.to, 2026-04-13 Context: Klaassen is general manager of Cora (Every’s AI email assistant) and creator of the compound-engineering methodology (14,000+ GitHub stars).

Summary

Klaassen spent three months trying to make agent swarms work - multiple AI agents coordinating with each other, deciding what to work on, producing output he hadn’t shaped. The experiment failed: when 10 agents finished simultaneously, he had 10 results to evaluate without enough context to trust any of them. The insight: “AI agents don’t have a speed limit, but the person managing them still does.”

He stopped looking for a smarter orchestration layer and realized the real agent was something he already had - a project folder. A folder with a CLAUDE.md/AGENT.md file, skill definitions, and context accumulated through months of compound-engineering makes a generalized model into a specialist.

The Folder-as-Agent Pattern

An agent is defined as: a model with enough context so you don’t have to re-explain everything each time you open the chat.

The ~/cora/ folder demonstrates the layers:

  • Conventions and standards - CLAUDE.md covers Rails conventions, deploy workflows, database patterns
  • Institutional knowledge - docs/developer-docs/ holds architecture reports, email processing pipeline, assistant system design
  • Operational memory - docs/runbooks/ and docs/investigations/ capture patterns from real incidents
  • Specialized agents - .claude/agents/ holds refined specialists: reviewers, planners, component creators

Reading order for new agents: CLAUDE.md first, then architecture document, then assistant system report, then assistant prompt, then component creator agent.

Key Claims

  • Klaassen runs 44 folder-agents across multiple projects, each in a specialized folder built over months
  • A dispatch layer routes between the 44 agents
  • Claude Code Skills - files that give the model specific capabilities - are an example of the folder-as-agent structure. People were already writing markdown instruction files and dropping them into project directories before anyone called them “skills”
  • Parallel Claude Code sessions work when each agent has a clear task and a human directs work; swarms where agents coordinate with each other fail
  • The same model (mostly Claude Opus 4.6, also GPT 5.4 and Gemini Pro 3.1) becomes a completely different agent when pointed at different folders

Relationship to Other Concepts

The folder-as-agent pattern is closely related to the agentic-constitution concept from YB - both treat structured markdown files as the primary interface between human intent and AI capability. The key difference: Klaassen’s pattern emphasizes project-level accumulated context (codebases, runbooks, incident history), while YB’s constitution emphasizes personal-level context (identity, goals, work style).

Both patterns converge on the idea that the value is in the accumulated context, not the model itself.

Counter-arguments

  • 44 agents sounds impressive but the article doesn’t describe what most of them do or how the dispatch layer works (clipping truncated)
  • The “folder is the agent” framing is essentially “give the model good context” - a well-known principle repackaged as a methodology
  • The failure of agent swarms may reflect the state of tooling in early 2026, not a fundamental limitation
  • No discussion of when folder-agents fail or produce poor output

Data gaps

  • Article clipping is truncated - likely missing sections on the dispatch layer, specific agent examples, and conclusions
  • No metrics on productivity gains from the 44-agent setup vs fewer agents
  • The dispatch layer between agents is mentioned but not described
  • No comparison with other orchestration approaches that did work
