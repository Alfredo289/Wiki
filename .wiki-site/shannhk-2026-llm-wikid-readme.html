Summary

llm-wikid is an open-source project by shann-holmberg that implements andrej-karpathy's llm-wiki-pattern as a ready-to-clone Obsidian vault. The core idea: you dump raw sources (tweets, articles, PDFs, ideas) into a raw/ folder, and an AI agent compiles them into structured, cross-referenced wiki pages in wiki/.

Key Claims

This is not RAG. The system explicitly distinguishes itself from retrieval-augmented generation. RAG re-derives answers every time by chunking documents and running vector search. This system pre-compiles knowledge into structured wiki pages, keeps them current, and pre-builds cross-references. At ~100 articles / ~400K words, Karpathy found this outperforms RAG for Q&A.

The compound loop. Every query answer gets filed back into the wiki, so the next query has more to work with. The system grows smarter with use, not just with new source material.

Agent-agnostic. Works with any agent that can read markdown and run shell commands - Claude Code, OpenClaw, Hermes, Codex, or custom setups.

Ingest Pipeline

The system processes sources in phases:

  1. Sort clippings - raw/clippings/ is an inbox. Files get moved to the right subfolder (tweets to bookmarks, articles to articles, PDFs to papers)
  2. Resolve - URLs get fetched via the appropriate tool (yt-dlp for YouTube, scrapling for web, X API for tweets)
  3. Media extraction - Images downloaded and analyzed, video transcripts extracted
  4. Classify and compile - Type-specific extraction (transcripts get speaker attribution, papers get method/findings, articles get thesis/arguments)
  5. Cross-link and bias check - Wikilinks, counter-arguments, data gaps on every page

Quality Controls

  • Bias checks: Counter-arguments and data gaps sections on every page
  • Validation gate: explored: false on all AI-created pages, only human sets true
  • Confidence levels: high/medium/low/uncertain based on source quality
  • Source tracing: Every claim links to a specific source page
  • No blind links: Every wikilink resolves to a page or stub

Tool Ecosystem

Tool Purpose
qmd Local hybrid BM25/vector search with LLM re-ranking
obsidian Vault UI, graph view, Dataview queries
yt-dlp YouTube transcript extraction
scrapling Web scraping
Obsidian Web Clipper Browser-to-vault clipping

Scaling Strategy

File-based with index scanning up to ~300 pages, then qmd as primary search, then PostgreSQL/Supabase at 500+.

Counter-arguments

  • Pre-compiled knowledge requires re-processing when understanding changes, unlike RAG which always pulls fresh chunks
  • The “outperforms RAG” claim comes from Karpathy’s personal experience at ~100 articles, not a controlled benchmark
  • Agent-maintained wikis can accumulate confident-sounding but wrong information if bias checks are superficial
  • The compound loop could also compound errors if early sources are flawed

Data gaps

  • No performance benchmarks comparing this approach to RAG systems at various scales
  • Unknown how well the pattern works beyond single-user knowledge bases (team/collaborative use)
  • No data on how much human curation time is actually needed to keep quality high
  • The “not RAG” framing may be a false dichotomy - hybrid approaches could combine both
