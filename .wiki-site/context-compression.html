Strategies for reducing the token cost of persistent context in LLM agent workflows. As projects accumulate context files (CLAUDE.md, memory, documentation), token consumption grows with every session. Context compression addresses this without sacrificing the benefits of compound-engineering.

Two Approaches

File-level compression

Rewrite context files into more concise language while preserving semantic meaning. Caveman Compress achieves ~46% token reduction by compressing CLAUDE.md and memory files, preserving code blocks, URLs, and structural elements while condensing prose.

Tradeoff: Compressed files are harder for humans to read and maintain. The compression itself requires an API call.

Retrieval-level compression (progressive disclosure)

Instead of loading all context at once, provide a layered retrieval system that surfaces only relevant context on demand. Claude-Mem implements a 3-layer system (search → timeline → full details) claiming ~10x token savings.

Tradeoff: Requires running additional infrastructure (databases, HTTP services). Effectiveness depends on how selectively the agent actually retrieves.

Why It Matters

Token costs are a direct function of context size. For practitioners running multiple folder-as-agent projects with accumulated context, costs compound. A folder-agent with a rich CLAUDE.md, documentation, and operational memory might consume thousands of tokens just for startup context before any actual work begins.

Context compression is the efficiency counterpart to compound-engineering: compound engineering says accumulate context for better results; context compression says manage that accumulated context efficiently.

Counter-arguments

  • For small projects, the overhead of compression tools may exceed the savings
  • CLAUDE.md files are typically small (1-3K tokens) - compressing them saves pennies
  • The real cost driver is usually conversation length, not startup context
  • “Just write concise CLAUDE.md files” may be simpler than adding tooling

Data gaps

  • No head-to-head comparison of these two approaches on the same project
  • No data on the crossover point where compression tools pay for themselves
  • Unknown whether compressed context degrades agent performance vs full context
  • Both tools are early-stage - no longitudinal usage data
